## 🧭 项目骨架认知（先对齐再分档）

源表覆盖五阶段，每阶段下若干工作项、每个工作项 4-6 个细节字段。**飞书化的关键判断**是：哪些信息适合放进**多维表格**（结构化、可查询、可关联），哪些适合放**文档**（叙述性、版本化），哪些适合放**任务/审批/日历**（带状态、带时限、带审批人），哪些只在**群**里走（沟通态、临时态）。

## 🔴 P0 — 项目启动必须先拍板（不定就动不了）

### 1. 数据中心载体：一团一档放哪里？

> 这是整个系统的"主键"。所有阶段都围绕"一个团"展开，源表里五阶段全部字段都需要挂在某个团 ID 下。

- [x] A. 飞书多维表格（Base）作为唯一数据底座 —— 主表「团组」+ 关联子表「成员名单」「供应商」「行程节点」「收付款」「反馈」，用关联字段串起来。优点：结构化、可视图、可触发自动化、可分享行链接。缺点：长文本不友好（如行程描述、研学手册大纲）。

- [ ] B. Base 存结构化字段 + 飞书文档存叙述性内容 —— Base 主存"团号/时间/人数/价格/状态"等字段；行程方案、研学手册大纲、复盘小结挂飞书文档，Base 里只存文档链接。

- [ ] C. 全部用文档目录组织 —— 一个团一个文档目录，里面分子文档。优点：自由度高。缺点：查不动、统计不动，违反"一团一档可检索"初衷。

- [ ] D. 外部数据库 + 飞书只做前端展示 —— 数据落在 PostgreSQL/Notion，飞书只发通知。重，先不做。

> **背景**：源表里既有"人数/价格/出发时间/团号"这种典型结构化字段，又有"行程方案/研学手册/接待小结"这种长文，单一载体都覆盖不了。

### 18. 飞书版本选择与采购预算

> 这套 SOP 能不能全落到飞书，关键看版本能力上限。先用免费版起跑，到达瓶颈再升级最划算。

**18.1 官方公开价（截至 2026-05，参考官方帮助中心 + 第三方汇总）**

| 版本 | 价格（人/月） | 人数上限 | 关键能力差异 |
|-|-|-|-|
| **标准版（免费）** | ¥0 | 100 人 | 基础 IM/文档/Base/日历/任务/审批；Base 单表 5 万行；自动化 1000 次/月；外部协作受限 |
| **商业版** | ¥50/人/月 | 500 人 | Base 行数与自动化次数大幅放开；外部联系人开放；多级审批；表单提交量放开 |
| **专业版** | ¥80/人/月 | 500 人 | 商业版基础上 + 更高 API 配额、应用引擎/妙搭额度、云空间扩容 |
| **旗舰版** | ¥120/人/月 | 不限 | 安全/合规/审计；专属客服；适合需要等保 / 数据驻留的场景 |
| **企业版** | 议价 | 不限 | SSO、企业级权限、私有化部署选项；研学操作团队基本用不到 |

> ⚠️ 价格仅供决策参考，实际以飞书官方报价为准；通常按年付费、按席位结算，可能有阶梯折扣。

**18.2 估算公式（用来心算预算）**

- 标准版：¥0（100 人内随便用）
- 商业版年费 ≈ **席位数 × ¥50 × 12** = 10 人团队约 ¥6,000/年；20 人约 ¥12,000/年
- 专业版年费 ≈ **席位数 × ¥80 × 12** = 10 人约 ¥9,600/年；20 人约 ¥19,200/年
- 关键：**只为真正用系统的内部人买席位**，外部地接/学校走"外部联系人/外部群"不占席位

**18.3 走哪条路？（单选）**

- [x] A. 先免费版跑 MVP，碰到瓶颈再升（推荐）—— 至少能跑 2-3 个月真实团组数据，碰到的第一堵墙大概率是自动化次数或外部协作

- [x] B. 直接上商业版（¥50/席）—— 团队已经决定长期用，省得后面迁移，预算可接受

- [ ] C. 直接上专业版（¥80/席）—— 预判要做表单收集 / 应用引擎 / 大量 API 调用

- [ ] D. 现在不定，等 MVP 跑完看数据再决策

> **决策备注**：

**18.4 预判会触发付费的功能（勾上表示"这条功能我必须要"，全部勾完就基本要商业版起步）**

- [x] 外部地接 / 组团方进飞书群协作（不占内部席位）

- [x] 多级 / 条件分支审批（如报价金额>X 走总监审批）

- [ ] Base 自动化每月超 1000 次（每团约 10-20 次触发，月 50 团就超了）

- [x] 给学校 / 家长发 Base 表单大量收集成员信息

- [ ] 用妙搭 / 应用引擎做对客 H5 或内部小工具

- [x] 单团成员名单 + 历史团组累计超 5 万行

- [ ] 影像/手册存储超 10GB × 内部人数 的云空间池

- [ ] 旅游局电子合同 / 教委系统 API 对接（高 API 配额需求）

**18.5 飞书做不了 / 不该用飞书做的部分（仅作认知对齐，无需勾选）**

- 对家长 / 学生（C 端）触达 —— 走微信，飞书 toC 不适合
- 旅游局电子合同、教委上报、税务发票 —— 外部强监管系统，飞书只能挂链接 + 存截图
- 企业小程序收款 —— 走支付通道，钱流不进飞书
- 几年期影像归档（TB 级）—— 飞书云空间贵，建议接 OSS/S3
